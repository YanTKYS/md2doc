# 後継版設計提案: 箇条書き処理の再設計とテスト基盤

v0.3.0 で発生した段落消失問題を踏まえ、有識者から後継版（v0.4.0 以降）に向けた
設計提案が提出された。本ドキュメントはその提案内容と、現実装に基づく妥当性評価をまとめる。

- 本ドキュメントは **提案・検討中の方針** を記録するもので、確定仕様ではない。
- 実装済みの事実・既知の挙動メモは `docs/dev-notes-word-interop.md` を参照。

---

## 1. 提案の要旨

1. v0.3.0 の 4 フェーズ方式（Parse → WriteAllText → ApplyStyles → InsertTables）は維持する
2. 失敗した実装パターンを **禁止ルール** として明文化する
3. 箇条書きを Word ネイティブのリスト機能に依存しない方式へ切り替える
4. 順序付きリストを内部モデルに加える
5. `.docx` の `word/document.xml` を直接検査する回帰テストを追加する
6. Markdown パーサと Word COM レンダラを分離し、Word 不要環境で単体テスト可能にする

---

## 2. 禁止ルール（提案）

v0.3.0 開発中に失敗が確認された以下のパターンを後継版で再採用しない。

| 禁止項目 | 失敗理由 |
|----------|----------|
| `Paragraphs.Add()` → Style 適用 → `Paragraphs.Add()` の逐次処理 | Word 内部再計算により段落参照がずれ、段落消失が起きる |
| `Paragraphs.Count` を使った末尾段落参照 | `ApplyBulletDefault()` が継続段落を暗黙追加し Count がずれる |
| `ApplyBulletDefault()` の使用 | トグル動作。既存リスト書式を除去する場合がある |
| `para.Range.Style = -1` のみでリスト書式を解除 | 直接適用された ListFormat は消えない |
| `Range.Style` 適用直後の段落生成 | 内部再計算ペンディング中に参照を取ると不安定 |

---

## 3. 箇条書き再設計（提案）

### 案 1: 安定モード — 記号 + インデント

Word のリスト機能を使わず、通常段落として出力する。

```
•   アプリを起動する
•   Markdownファイルを選択する
```

- 段落スタイルは通常段落（`wdStyleNormal`）
- 左インデント・ぶら下げインデントで見た目を整える
- 箇条書き記号文字は md2doc 側で固定（`•` など）

**メリット**
- Word のリスト自動伝播・テンプレート依存を完全に避けられる
- 段落消失・交互消失の再発リスクが消える
- 見た目を md2doc 側で完全に制御できる
- 実装・テストが単純

**デメリット**
- Word 上は「本物の箇条書き」ではなく通常段落になる
- Word のリスト編集機能・アウトライン・アクセシビリティとは連動しない

### 案 2: Word リストモード — 専用 ListTemplate

`wdStyleListBullet` への全面依存を避け、md2doc 専用の ListTemplate を作成して適用する。

- `BlockKind.Bullet` の連続範囲を検出
- 連続範囲ごとに Range を構築し、専用 ListTemplate を適用
- 非リスト段落には安全策として `RemoveNumbers()`

**メリット**
- Word 上で本物の箇条書きとして扱える
- 後から Word で編集しやすい
- 順序付きリストへの拡張も同じ枠組みで可能

**デメリット**
- COM 操作が複雑化する
- Word バージョン差・テンプレート差の影響を受けやすい
- 自動補正・伝播の問題が完全には消えない

### 推奨

設定で切り替え可能にし、初期値は **案 1（安定モード）** とする。

```
箇条書き出力方式:
  - 安定モード: 記号 + インデント（初期値）
  - Word リストモード: Word のリスト機能を使用
```

### 評価メモ

案 1 は「Word ネイティブな箇条書き」を捨てるトレードオフがある。
md2doc の利用シーン（Word 上で後編集するか、生成物をそのまま使うか）によって
最適解が変わるため、採用前に実利用パターンを確認することが望ましい。

## 4. 順序付きリストの内部モデル化（提案）

現在 `1.` / `2.` は `BlockKind.Paragraph` として扱われている。
後継版では内部モデルに加える。

```csharp
private enum BlockKind
{
    Empty, Paragraph, Heading,
    UnorderedListItem,   // 旧 Bullet
    OrderedListItem,     // 新規
    Hr, PageBreak, Table
}

private sealed class Block
{
    public BlockKind Kind;
    public string Text = "";
    public int HeadingLevel;
    public int ListLevel;          // ネスト用（初期は 0 のみ）
    public int? OrderedNumber;     // OrderedListItem 用
    public bool StartsNewList;     // 連続範囲検出補助
    // ...
}
```

初期実装ではネストなし（`ListLevel = 0` のみ）で良いが、
内部モデルだけは将来の拡張に備えた形にしておく。

## 5. 回帰テスト方針（提案）

段落消失は Word 上の見た目だけでは検知漏れが起きやすい。
生成された `.docx` を ZIP として展開し、`word/document.xml` を直接検査するテストを用意する。

### 必須テストケース

```markdown
# タイトル
## 起動方法
- アプリを起動する
- Markdownファイルを選択する
本文1
- 箇条書きA
- 箇条書きB
本文2
```

検証項目:

- すべてのテキストが `document.xml` に存在する
- テキストの順序が Markdown と一致している
- 見出し直後の箇条書き 1 行目が消えていない
- 箇条書きが奇数行/偶数行のみに減っていない
- 箇条書き後の通常段落がリスト化されていない

### 追加テストケース

```markdown
## 表の前
- 箇条書き1
- 箇条書き2

| A | B |
|---|---|
| 1 | 2 |

- 表の後の箇条書き
本文
```

テーブル前後の箇条書きは、インデックスシフト・リスト伝播・テーブル挿入が絡む
高難度ケース。回帰検知に特に有効。

## 6. 実装分割（提案）

現在の `WordInteropConverter` は Markdown 解析・Word 書き込み・スタイル適用・
テーブル挿入・ヘッダーフッター処理を一体で持っている。後継版では以下に分割する。

```
MarkdownParser
  入力: Markdown 文字列
  出力: DocumentModel（List<Block>）
  Word COM 非依存 → Word なしで単体テスト可能

DocumentModel
  Heading / Paragraph / ListItem / Table / PageBreak など

WordComRenderer
  入力: DocumentModel
  出力: .docx ファイル
  Word COM 依存 → 統合テストでカバー

DocxInspectorTestHelper
  入力: .docx ファイル
  出力: document.xml 解析結果
  回帰テスト用
```

`ParseBlocks` は現状でも Word COM に触れない純粋関数なので、
分離コストは小さい。

## 7. 推奨ロードマップ

| Step | 内容 | 目的 |
|------|------|------|
| 1 | 4 フェーズ方式の維持と禁止ルール明文化 | 既知の地雷を踏まない |
| 2 | 箇条書き安定モード（記号 + インデント）追加。初期値とする | リスト伝播問題の根治 |
| 3 | 順序付きリストの内部モデル化 | 機能拡張 + モデル整備 |
| 4 | docx XML ベースの回帰テスト追加 | 段落消失の自動検知 |
| 5 | Parser / Model / Renderer の分離 | テスト可能性の向上 |

## 8. 妥当性評価（md2doc 開発側）

提案内容を現実装と照らして評価した結果は以下。

| 項目 | 評価 | コメント |
|------|------|----------|
| 4 フェーズ方式の維持 | 妥当 | v0.3.0 で実証済み |
| 禁止ルールの明文化 | 妥当 | 失敗パターンは `dev-notes-word-interop.md` に記録済み。後継版で制約として再記述する価値あり |
| 案 1（記号 + インデント）標準化 | 条件付き妥当 | 安定性は最大化されるが Word ネイティブ箇条書きを失う。利用シーン次第 |
| 案 2（専用 ListTemplate）温存 | 妥当 | 互換モードとして残す位置づけは正しい |
| 順序付きリストの内部モデル化 | 妥当 | 現状は段落扱いのため拡張余地として自然 |
| docx XML 検査 | 強く妥当 | 段落消失系は見た目テストでは検知できない。最優先で導入すべき |
| Parser/Model/Renderer 分離 | 妥当 | `ParseBlocks` は既に純粋関数化済みのため分離コスト小 |

### 残検討事項

- 案 1 を標準にした場合、Word のアウトライン表示・目次（TOC）連携で
  箇条書き項目が見出し階層下に正しくぶら下がるかは未検証
- 案 1 と案 2 の切り替え設定 UI の置き場所（オプションタブ新設 vs 既存設定への追加）
- docx XML 検査テストの実行環境（Word なしの CI でも動くべき）
